Three measured improvements

Optimisations

Each was benchmarked before and after. Numbers from the actual test suite.

01 · integer XOR
_xor_many fast path

Replacing byte-by-byte XOR with a single integer operation. int.from_bytes + ^ is one C-level call in CPython instead of 16 interpreter iterations over a loop.

Before · byte loop
6.59 μs
for b in zip(*values):
  acc ^= b
After · int XOR
1.10 μs
acc = int.from_bytes(v)
acc ^= ...
↑ 5.9× faster · 20 000 runs
02 · hash backend
BLAKE2b vs SHA-256

One-line swap in utils.py: HASH_BACKEND = "blake2b". BLAKE2b has lower per-call overhead on short 16-byte inputs. Same 128-bit security level — measurably faster keygen.

↑ 1.18× faster keygen
03 · lazy chains
GenLazy / SignLazy

Eager Gen stores full chains: 18 × 256 × 16 = 73,728 bytes per key. GenLazy stores only 18 seeds (288 B) and recomputes on demand during signing. The right tradeoff for constrained devices.

Layer architecture

Five composable layers — each one independently tested, cleanly separated.

5

PRF Distributed HBS

Two-round protocol. Each trustee contributes PRF_R, PRF_Chain, PRF_Auth shares. Aggregator XOR-combines via _xor_many. No trustee learns the full signing key.

top layer
4

Shard Trustee API

High-level aggregator + trustee interface: ShardSetup, TrusteeSetup, AggregatorSign, AggregatorVerify.

API layer
3

Stateful Hash-Based Signatures

Manages a Merkle tree of OTS keys, tracks used indices, generates authentication paths for the non-distributed baseline.

state layer
2

Merkle Tree

Commits to all OTS public keys. MT_Construct O(D), MT_MakePath O(D + log D), MT_Verify O(log D).

tree layer
1

Winternitz OTS + Lazy variant

Hash chains of length 256 (W=8). Gen builds full chains; GenLazy stores only seeds. Both produce identical public keys.

OTS layer
0

Tweakable Hash — SHA-256 / BLAKE2b

Domain-separated via hash_lms(tweak, ...). Swap backends in one line; benchmarks show BLAKE2b 1.18× faster for short inputs.
